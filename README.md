# TrustVerify

TrustVerify is a Python-based CLI project developed collaboratively by our team to verify **file integrity** and **file origin** using **SHA-256 hashing** and **RSA digital signatures**.

## Project Objective

In this project, we aimed to build a Command Line Interface (CLI) tool that allows a **Sender** to sign a file and a **Receiver** to verify both its integrity and authenticity.

## Project Scope

As part of this project, we implemented the following features:
- generating SHA-256 hashes for files
- creating a `metadata.json` manifest for a directory
- checking files against the manifest to detect tampering
- generating RSA public/private key pairs
- signing the manifest
- verifying the signature using the sender’s public key

## Why Hashing Alone Is Not Enough to Prove Identity

Hashing algorithms such as SHA-256 are useful for verifying **integrity**. They allow us to check whether a file has been modified by comparing its current hash value with the original one. If the values match, the file is considered unchanged.

However, hashing alone does **not** provide **authentication**. An attacker may intercept a file, modify its content, and generate a completely new valid hash for the altered version. In that case, the receiver may observe a correct hash-file match, but still cannot determine whether the file truly came from the original sender.

For this reason, hashing proves that content matches a given hash value, but it does not prove **who created or sent** that content. Therefore, hashing alone is insufficient to establish identity or trust the origin of a file.

## How the Private/Public Key Relationship Ensures Non-Repudiation

In asymmetric cryptography, each user has a **private key** and a **public key**. The private key is kept secret by its owner, while the public key can be shared publicly.

When a sender digitally signs a file or its hash, the signature is created using the sender’s **private key**. The receiver then verifies that signature using the sender’s **public key**. If the verification succeeds, it demonstrates that the signature could only have been generated by the holder of the matching private key.

This provides **non-repudiation**, meaning that the sender cannot later deny having signed the data. Since only the sender should possess the private key, a valid digital signature strongly connects the signed content to that specific sender. In this way, digital signatures provide both **authenticity** and **accountability**, which hashing alone cannot provide.

## Demo Summary

In our live demonstration, we first create the manifest and sign it. Then, we deliberately modify one of the files. When verification is performed again, the tool detects the tampering and returns a **Verification Failed** result.
